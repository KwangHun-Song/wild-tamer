# 코드 리뷰 컨벤션

## 개요

시스템 구현 워크플로우의 각 단계에서 산출물(문서, 코드)의 품질을 검증하기 위한 리뷰 규칙을 정의한다.

## 리뷰 시점

리뷰는 워크플로우 단계 완료 시 진행하며, 리뷰 대상에 따라 범위가 달라진다.

| 리뷰 시점 | 리뷰 대상 | 산출물 |
|-----------|-----------|--------|
| 컨셉/설계 완료 후 | `concept.md`, `design.md` | `review/concept_design_review.md` |
| 구현 완료 후 | 구현 코드 + 설계 문서 대비 | `review/code_review.md` |
| 컨셉/설계/구현 동시 | 문서 + 코드 전체 | `review/concept_design_review.md` (통합) |

### 리뷰 산출물 위치

리뷰 문서는 해당 시스템 폴더 내 `review/` 하위에 저장한다.

```
System/SceneFlow/
├── concept.md
├── design.md
├── implementation_plan.md
├── review/
│   ├── concept_design_review.md
│   └── code_review.md
└── trd.md
```

## 리뷰 문서 구조

### 컨셉/설계 리뷰 (`concept_design_review.md`)

```markdown
# {시스템명} - 컨셉/설계 리뷰

## 컨셉 문서 리뷰
### 작업 컨벤션 충족도    ← 컨벤션 기준 체크
### 긍정적인 점           ← 잘된 부분 먼저
### 개선 제안             ← 번호 + 중요도 표기

## 설계 문서 리뷰
### 작업 컨벤션 충족도
### 긍정적인 점
### 개선 제안

## 종합 평가              ← 항목별 등급 + 우선 보강 요약
```

### 코드 리뷰 (`code_review.md`)

```markdown
# {시스템명} - 코드 리뷰

## 리뷰 대상 파일         ← 파일 목록 테이블
## 긍정적인 점
## 이슈                   ← 번호 + 중요도 + 코드 인용
## 종합 평가
```

#### 코드 리뷰 파일 분할

시스템 규모가 커서 리뷰 대상 파일이 많은 경우, **주제별로 분할**하여 작성할 수 있다.

```
review/
├── code_review_01_common.md    # 공통 엔티티 레이어
├── code_review_02_player.md    # 플레이어 관련
├── code_review_03_squad.md     # 스쿼드 시스템
└── ...
```

- 파일명은 `code_review_{번호}_{주제}.md` 형식을 따른다.
- 각 파일의 구조는 단일 `code_review.md`와 동일하게 유지한다.
- 분할 여부는 리뷰 대상 파일 수(10개 이상 기준)와 주제 독립성을 기준으로 판단한다.

### 통합 리뷰 (문서 + 코드)

컨셉/설계와 구현이 함께 리뷰되는 경우, 하나의 문서에 통합한다.

```markdown
# {시스템명} - 컨셉/설계/구현 리뷰

## 컨셉 문서 리뷰
## 설계 문서 리뷰
## 구현 코드 리뷰
## 종합 평가
```

## 리뷰 항목

### 컨셉 문서 체크리스트

작업 컨벤션에서 정의한 항목을 기준으로 충족 여부를 판정한다.

| 항목 | 설명 |
|------|------|
| 시스템의 목적과 역할 | 이 시스템이 무엇을 하는지 명확한가 |
| 유저 관점의 동작 설명 | 사용자(개발자 포함) 관점에서 동작 흐름이 설명되어 있는가 |
| 레퍼런스 자료 | 참고한 자료(이미지, 영상, 타 게임 사례 등)가 있는가 |
| 다른 시스템과의 관계 | 기존 시스템과의 의존/협력 관계가 명시되어 있는가 |

### 설계 문서 체크리스트

| 항목 | 설명 |
|------|------|
| 핵심 클래스 및 인터페이스 구조 | 주요 타입의 책임과 관계가 정의되어 있는가 |
| 클래스 간 의존 관계 및 데이터 흐름 | 의존성 방향과 데이터 전달 방식이 명확한가 |
| 사용할 디자인 패턴 | 적용한 패턴이 있다면 명시하고 선택 근거가 있는가 |
| 외부 시스템과의 인터페이스 정의 | 다른 모듈/시스템과의 연동 방식이 정의되어 있는가 |

### 코드 리뷰 체크리스트

| 항목 | 설명 |
|------|------|
| 설계 일관성 | 구현이 설계 문서와 일치하는가. 차이가 있다면 개선인가 누락인가 |
| 코딩 컨벤션 준수 | `csharp_coding_convention.md`의 명명 규칙, 코드 스타일을 따르는가 |
| 에러 처리 | 예외 상황에 대한 처리가 적절한가. 특히 비동기 코드의 에러 전파 |
| 캡슐화 | 접근 제한자가 적절한가. 내부 구현이 불필요하게 노출되지 않았는가 |
| 테스트 존재 | 유닛 테스트가 작성되어 있는가. 핵심 로직이 검증 가능한가 |

## 충족도 표기

컨벤션 충족 여부는 다음 기호로 표기한다.

| 기호 | 의미 |
|------|------|
| O | 충족 |
| △ | 부분 충족 (보완 필요) |
| X | 미충족 |

## 이슈 중요도

개선 제안 및 이슈에는 중요도를 표기하여 우선순위를 명확히 한다.

| 중요도 | 기준 | 조치 |
|--------|------|------|
| 높음 | 버그, 크래시, 보안, 데이터 유실 가능성 | 구현 반영 후 다음 단계 진행 |
| 중간 | 확장성, 유지보수성, 누락된 기능 | 다음 단계 진행 가능하나 기록하여 추적 |
| 낮음 | 네이밍, 문서 불일치, 사소한 개선 | 편의에 따라 반영 |

## 종합 평가

### 등급 기준

| 등급 | 기준 |
|------|------|
| A | 우수. 개선 사항이 거의 없거나 낮음 수준만 존재 |
| B+ | 양호. 구조적으로 잘 되어 있으나 중간 수준의 보완 필요 |
| B | 보통. 핵심은 갖추었으나 중요한 보완이 필요 |
| C | 미흡. 높음 수준의 이슈가 다수 존재하여 수정 후 재리뷰 필요 |

### 종합 평가 테이블

리뷰 문서 하단에 항목별 등급과 우선 보강 사항을 정리한다.

```markdown
## 종합 평가

| 항목 | 점수 | 설명 |
|------|------|------|
| 구조 설계 | A | ... |
| 인터페이스 설계 | B+ | ... |
| 구현 품질 | B | ... |
| 컨벤션 준수 | A- | ... |

### 우선 보강이 필요한 N가지

1. ...
2. ...
3. ...
```

## 리뷰 작성 원칙

### 긍정적인 점을 먼저 기술한다

잘된 부분을 먼저 짚어 설계 의도를 확인하고, 이후에 개선 제안을 기술한다.

### 이슈에는 근거와 대안을 함께 제시한다

문제만 지적하지 않고, 왜 문제인지(근거)와 어떻게 개선할 수 있는지(대안)를 함께 기술한다.

```markdown
// 좋은 예
#### 1. SceneLauncher의 async void 예외 처리 부재 (중요도: 높음)

**파일**: `SceneLauncher.cs:8-12`

`async void`에서 예외가 발생하면 Unity의 기본 예외 처리기로 전달되어
앱이 크래시할 수 있다.

제안:
    try-catch로 감싸서 Facade.Logger로 에러를 기록한다.

// 나쁜 예
#### 1. 에러 처리가 없음
async void는 위험하다. 고쳐야 한다.
```

### 코드 인용 시 파일 경로와 라인을 명시한다

이슈와 관련된 코드를 인용할 때는 파일 경로와 라인 번호를 포함하여 위치를 특정한다.

```markdown
**파일**: `SceneStateMachine.cs:36-49`

코드블록으로 해당 코드를 인용한다.
```

### 설계 문서와 구현의 차이를 명확히 구분한다

차이가 발견된 경우, 구현이 더 나은 방향인지 설계 의도에서 벗어난 것인지를 판단하여 기술한다.

```markdown
| 항목 | 설계 문서 | 실제 구현 | 판단 |
|------|-----------|-----------|------|
| 메서드 분리 | 2개 | 1개로 통합 | 구현이 개선된 형태 |
```
