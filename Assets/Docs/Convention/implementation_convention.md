# 구현 컨벤션

## 구현 계획 문서 작성 규칙

구현 계획 문서(`implementation_plan.md`)는 다음 내용을 포함한다.

### 1. 기술 스택

구현에 사용하는 주요 라이브러리, 프레임워크, Unity 패키지 등을 명시한다.

### 2. 시스템 설계

설계 문서(`design.md`)를 기반으로 구현 관점에서 구체화한다.

- 구현할 클래스/파일 목록
- 클래스별 책임과 핵심 메서드
- 의존성 관계

### 3. 구현 순서

작업 항목을 단계별로 나열하되, **병렬 실행 가능 여부를 명시**한다.

```markdown
#### Step 1: 기반 작업
- [ ] 작업 A (선행 작업 없음)

#### Step 2: 핵심 구현 [병렬 가능]
- [ ] 작업 B (A 완료 후)
- [ ] 작업 C (A 완료 후)
- [ ] 작업 D (A 완료 후)

#### Step 3: 통합
- [ ] 작업 E (B, C, D 모두 완료 후)
```

### 4. 테스트 계획

- 유닛 테스트 대상 및 방법
- 수동 검증 항목

---

## 구현 실행 규칙

### 병렬 처리

- 구현 순서에서 `[병렬 가능]`으로 표시된 항목들은 **서브 에이전트를 사용하여 병렬로 실행**한다.
- 해당 Step의 태스크가 **1개뿐이면 서브 에이전트 없이 직접 처리**한다.

### 테스트

- 모듈 구현 시 `Tests/` 폴더를 생성하고 **유닛 테스트를 작성**한다.
- 테스트 asmdef 이름은 `{ModuleName}.Tests` 형식을 따른다.
- 유닛 테스트 작성 후 **Unity Test Runner (EditMode)로 실행하여 통과를 확인**한다.
- 유닛 테스트로 검증하기 어려운 항목은 수동 검증 항목으로 기록한다.

### 커밋 단위

- 구현 순서의 **Step 단위** 또는 **의미 있는 작업 단위**로 커밋한다.
- 커밋 전 컴파일 에러가 없는지 확인한다.
- 테스트가 있는 경우 테스트 통과 후 커밋한다.

### 설계 변경

- 구현 중 설계 변경이 필요하면 **설계 문서(`design.md`)를 먼저 업데이트**한 후 코드를 수정한다.
- 구현 계획에서 벗어나는 변경이 발생하면 **구현 계획 문서도 함께 업데이트**한다.
