# 커밋 컨벤션

## 커밋 메시지 템플릿

```
커밋 내용 요약
- 커밋 상세내용 1
- 커밋 상세내용 2
```

### 작성 규칙

- **요약**: 변경 사항을 한 줄로 명확하게 설명한다.
- **상세내용**: 무엇을 왜 변경했는지 구체적으로 기술한다.
- 요약과 상세내용 사이에 빈 줄을 둔다.

### 예시

```
플레이어 이동 로직 구현
- Rigidbody2D 기반 상하좌우 이동 처리 추가
- 이동 속도를 Inspector에서 조절할 수 있도록 SerializeField 적용
```

## 커밋 단위

- 커밋은 **하나의 구분되는 작업 단위**여야 한다.
- 서로 관련 없는 변경 사항을 하나의 커밋에 섞지 않는다.
- 커밋 하나만 되돌려도 기능 단위로 깔끔하게 롤백할 수 있어야 한다.

### 좋은 예

- 새 기능 하나를 추가하는 커밋
- 하나의 버그를 수정하는 커밋
- 리팩토링만 포함된 커밋

### 나쁜 예

- 버그 수정 + 새 기능 추가 + 코드 정리를 한 커밋에 포함
- "여러 가지 수정"처럼 범위가 불분명한 커밋

## 커밋 메시지 작성 원칙

- **"무엇을"과 "왜"**에 집중한다. "어떻게"는 코드가 설명한다.
- 요약은 간결하되 변경 의도가 드러나도록 작성한다.
- 모호한 표현(`수정`, `업데이트`, `변경`)만 단독으로 쓰지 않는다. 대상을 함께 명시한다.

```
# Good
플레이어 체력 회복 로직 버그 수정

# Bad
버그 수정
```

## 커밋 빈도

- 작업이 의미 있는 단위로 완료될 때마다 커밋한다.
- 너무 작은 단위(오타 하나)를 남발하지 않되, 큰 작업을 한 번에 몰아서 커밋하지도 않는다.
- 컴파일이 되지 않는 상태로 커밋하지 않는다.

## 커밋 전제 조건

커밋은 다음 조건을 **모두 충족**한 상태에서만 진행한다.

1. **작업 완료**: 해당 작업 단위의 구현이 완료되었다.
2. **테스트 통과**: 유닛 테스트가 있는 경우 모든 테스트가 통과한다.
3. **오류 없음**: 컴파일 에러, 런타임 에러, 콘솔 경고가 없다.
4. **실행 확인**: 에디터에서 정상 실행되며 기존 기능에 영향이 없다.

작업이 진행 중이거나, 테스트가 실패하거나, 오류가 남아있는 상태에서는 커밋하지 않는다.

## 스테이징 규칙

- `git add .`이나 `git add -A`처럼 **모든 변경 사항을 일괄 스테이징하지 않는다.**
- 작업한 내용을 파악하고, **해당 작업에 관련된 파일만 선별하여 스테이징**한다.
- 스테이징 전 `git status`와 `git diff`로 변경 내용을 확인하고, 작업 단위에 포함되지 않는 파일은 제외한다.

```bash
# Good - 작업한 파일만 선별 스테이징
git add Assets/Modules/Base/Runtime/Scripts/Notifier/Notifier.cs
git add Assets/Modules/Base/Tests/NotifierTests.cs

# Bad - 모든 변경 사항 일괄 스테이징
git add .
```

### 예외: 바이너리 에셋 대규모 정리

다음 조건을 **모두** 만족하는 경우 `git add -A` 일괄 스테이징을 허용한다.

1. 해당 커밋의 **변경 내용이 에셋 정리/이동/삭제 단일 목적**이다.
2. 코드(.cs) 변경이 **포함되지 않거나** 에셋 정리와 직접 연관된 최소한의 변경만 포함된다.
3. `git status`로 변경 파일 목록을 확인하고 **의도하지 않은 파일이 없음**을 검토한 후 진행한다.

## 커밋 전 체크리스트

- [ ] 작업 단위의 구현이 완료되었는가?
- [ ] 유닛 테스트가 모두 통과하는가?
- [ ] 컴파일 에러 및 런타임 에러가 없는가?
- [ ] 에디터에서 정상 실행되는가?
- [ ] 커밋 단위가 하나의 작업에 대응하는가?
- [ ] 커밋 메시지가 변경 내용을 명확히 설명하는가?
- [ ] 불필요한 파일(임시 파일, 로그 등)이 포함되지 않았는가?
