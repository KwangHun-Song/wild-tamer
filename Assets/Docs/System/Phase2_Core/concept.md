# Phase 2: 코어 시스템 - 컨셉

## 목적

Phase 1에서 구축한 프로젝트 베이스(데이터 관리, UI, 사운드, 오브젝트 풀링, 이벤트 시스템) 위에 게임의 핵심 플레이 루프를 구현한다. 플레이어가 월드를 탐험하고, 야생 몬스터와 자동 전투하고, 처치한 몬스터를 테이밍하여 부대를 확장하는 코어 루프를 완성하는 것이 목표다.

## 코어 루프

```
탐험 → 전투 → 테이밍 → 부대 확장 → 탐험 ...
```

### 유저 관점 플레이 경험

1. **탐험**: 플레이어가 조이스틱/키보드로 캐릭터를 이동시키면 테이밍한 부대원들이 대형을 유지하며 따라온다. 쿼터뷰 카메라가 플레이어를 추적하고, 이동할수록 전장의 안개가 걷히며 새로운 지역이 드러난다. 미니맵에서 탐색 현황을 실시간으로 확인한다.
2. **전투**: 야생 몬스터 무리와 조우하면 아군 부대와 적이 자동으로 교전한다. 타격 시 역경직과 화면 흔들림, 이펙트와 사운드가 어우러져 타격감을 제공한다.
3. **테이밍**: 몬스터를 처치하면 확률적으로 테이밍 판정이 발생한다. 성공 시 시각적 피드백과 함께 해당 몬스터가 아군 부대에 합류한다.
4. **부대 확장**: 부대원이 늘어날수록 군집의 규모가 커지고, 더 강한 몬스터와 보스에 도전할 수 있게 된다. 확장된 전력으로 미탐색 지역을 향해 다시 탐험에 나선다.

## 시스템 개요

### 2.1 플레이어 이동 및 입력

플레이어 캐릭터의 이동을 처리하고, 쿼터뷰 카메라가 플레이어를 부드럽게 추적한다. 조이스틱 또는 키보드 입력을 받아 2D 쿼터뷰 공간에서의 이동 방향을 결정한다.

### 2.2 군집 및 부대 시스템

NavMesh를 사용하지 않고 직접 구현한 길찾기로 부대원들이 플레이어를 따라다닌다. 부대원 간 겹침을 방지하고 대형을 유지하며 부드러운 군집 이동을 처리한다.

### 2.3 월드맵

타일맵 또는 프로시저럴 기반으로 맵을 생성하고, 지형과 장애물을 배치한다. 플레이어와 몬스터가 활동하는 게임 월드의 물리적 공간을 제공한다.

### 2.4 몬스터

일반 몬스터 3종(종별 이동 패턴, 공격 사거리 차별화)과 보스 몬스터 2종(장판, 돌진 등 특수 스킬 패턴)을 구현한다. 몬스터 랜덤 스폰과 야생 몬스터의 부대 형태 군집 이동을 포함한다.

### 2.5 자동 전투 시스템

적 감지, 자동 타겟팅, 아군/적 자동 교전 로직을 처리한다. 타격 판정과 데미지 계산, 체력 시스템 및 사망 처리를 담당한다.

### 2.6 테이밍

몬스터 처치 시 확률 기반으로 테이밍 판정을 수행한다. 성공 시 해당 몬스터를 아군 부대에 합류시키고, 시각적 피드백으로 테이밍 성공을 연출한다.

### 2.7 전투 연출

공격/피격 시 역경직(히트스톱), 화면 흔들림(카메라 셰이크), 타격 이펙트 및 사운드를 통해 전투의 손맛과 타격감을 제공한다.

### 2.8 전장의 안개 (Fog of War)

미탐색 지역을 어둡게 처리하고, 플레이어 이동에 따라 시야를 공개한다. 탐색한 영역과 미탐색 영역의 시각적 구분을 제공한다.

### 2.9 미니맵

화면 한쪽에 미니맵 UI를 표시하여 실시간 탐색 현황을 반영한다. 플레이어, 아군, 적의 위치를 아이콘으로 표시한다.

## 구현 순서

의존성을 기반으로 레이어별로 구현한다. 하위 레이어가 완성되어야 상위 레이어 구현이 가능하다.

### Layer 1 — 기반

| 시스템 | 이유 |
|--------|------|
| 2.1 플레이어 이동 및 입력 | 모든 시스템의 기준점. 입력과 카메라가 동작해야 이후 시스템 테스트 가능 |
| 2.3 월드맵 | 이동, 군집, 스폰 등 공간 기반 시스템이 참조할 맵 데이터 필요 |

### Layer 2 — 개체

| 시스템 | 이유 |
|--------|------|
| 2.2 군집 및 부대 시스템 | 플레이어와 맵이 있어야 군집 이동 구현/테스트 가능 |
| 2.4 몬스터 | 맵 위에 스폰되고 군집 로직을 공유하므로 맵과 군집 후 구현 |

### Layer 3 — 상호작용

| 시스템 | 이유 |
|--------|------|
| 2.5 자동 전투 시스템 | 아군(부대)과 적(몬스터)이 모두 존재해야 교전 로직 구현 가능 |

### Layer 4 — 결과 및 연출

| 시스템 | 이유 |
|--------|------|
| 2.6 테이밍 | 전투 결과(처치)에 의존. 전투 시스템 완성 후 구현 |
| 2.7 전투 연출 | 전투 이벤트를 수신하여 연출을 트리거하므로 전투 시스템 이후 |

### Layer 5 — 시야 및 UI

| 시스템 | 이유 |
|--------|------|
| 2.8 전장의 안개 | 코어 루프와 독립적인 시각 레이어. 플레이어 이동과 맵만 있으면 구현 가능하나 게임플레이 확인 후 조정이 필요하므로 후순위 |
| 2.9 미니맵 | 전장의 안개 및 개체 위치 정보를 종합하여 표시하므로 최후순위 |

## 기술적 고려사항

### 다수 개체 최적화

부대원과 몬스터가 동시에 다수 존재하므로 성능이 핵심 과제다. Phase 1의 오브젝트 풀링을 적극 활용하여 생성/파괴 비용을 줄이고, 공간 분할(그리드 또는 쿼드트리)로 탐색/충돌 범위를 제한한다. 화면 밖 개체는 업데이트 빈도를 낮추는 LOD 전략을 고려한다.

### NavMesh 미사용 군집 이동

길찾기를 직접 구현해야 하므로 알고리즘 설계가 중요하다. Boids 기반 군집 행동(정렬, 응집, 분리)에 플레이어 추적과 장애물 회피를 결합한다. 야생 몬스터도 동일한 군집 로직을 공유하여 코드 재사용성을 높인다.

### 2D 쿼터뷰

2D 스프라이트 기반 쿼터뷰 시점에서의 깊이 정렬(Y-sort 또는 레이어 기반)과 이동 방향 매핑을 고려해야 한다. 카메라는 직교(Orthographic) 투영을 사용하며 플레이어를 부드럽게 추적한다.

### 이벤트 기반 느슨한 결합

시스템 간 직접 참조를 최소화하고 Phase 1의 이벤트 시스템을 통해 통신한다. 전투→테이밍(처치 이벤트), 전투→연출(타격 이벤트) 등의 흐름을 이벤트 버스로 처리하여 시스템 간 결합도를 낮춘다.
